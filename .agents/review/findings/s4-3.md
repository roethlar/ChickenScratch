# s4-3: Attestation is not re-checked at the last safe point before the reset

**Severity**: MEDIUM — the check-to-reset window is microseconds (no fetch sits
inside it, unlike the old force-pull), but the plan (round 11) explicitly
requires the re-check "immediately before the hard reset" and the consequence
inside the window is silent overwrite.
**Status**: Verified
**Branch**: n/a — pre-commit fix folded into the slice-4 commit (see s4-1).
**Commit**: (slice-4 commit)

## Evidence
`crates/core/src/core/git.rs`: `force_resolve_merge` runs
`recovery.ensure_valid_for` at function entry, then opens the repository and
resolves the target object before `repo.reset`; `sync_abort_pull` and
`complete_merge` have the same entry-only shape.

## Predicted observable failure
A concurrent external edit lands after the entry check while the repo is
being opened / the object resolved; the reset (or staging) proceeds and
overwrites it without the mandated fail-closed refusal.

## What
Single entry-time attestation instead of entry + last-safe-point.

## Approach
Keep the entry check (it validates the root before any repo access) and add a
second `ensure_valid_for` immediately before the point of no return in all
three recovery operations: before the hard reset in `force_resolve_merge` and
`sync_abort_pull`, before staging in `complete_merge`. After the reviewer's
round-2 reopen, `complete_merge` opens its index BEFORE the re-check so no
fallible disk operation remains between the re-check and the mutation.

## Files changed
- `crates/core/src/core/git.rs` — re-attestation adjacent to the point of no
  return in the three recovery operations

## Guard proof
The window is internal and cannot be driven from the public API without a
mutating failpoint; the existing drift regressions
(`force_resolve_fails_closed_on_external_edits_after_confirmation`,
`..._when_a_different_merge_started`, `recovery_authority_fails_closed_when_
the_merge_ends_before_use`) pin the fail-closed behavior, and the code order
(re-check adjacent to reset/staging) is the review-verified structural fix.

## Coder dispute (if any)
None.

## Known gaps
The re-check narrows but cannot erase the TOCTOU window — that is inherent to
check-then-act on a shared filesystem; the plan's own wording ("last safe
point") accepts this.
