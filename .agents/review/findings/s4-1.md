# s4-1: Force resolution is not bound to the merge state the writer confirmed

**Severity**: HIGH — the one destructive recovery action can act on a different
merge than the one the writer saw and confirmed; silent data loss.
**Status**: Verified
**Branch**: n/a — admitted pre-commit against the uncommitted slice-4 working
tree; the fix folds into the slice commit (recorded deviation from the
per-finding-branch mechanic, which presumes committed code).
**Commit**: (slice-4 commit)

## Evidence
`ui/src/components/revisions/Revisions.tsx` (handleForceResolve) completes the
destructive confirmation dialog with no attestation captured;
`src-tauri/src/commands/git.rs` `force_resolve_merge_inner` acquires the
`RecoveryPermit` — which is what binds `MERGE_HEAD` + fingerprint — only when
the dispatch reaches Tauri. Plan round 13 requires capturing the attestation
"at confirmation time".

## Predicted observable failure
Conflict dialog sits open on merge A; another process/window aborts A and
starts merge B; the writer clicks "Take the incoming version" believing it
resolves A → the fresh permit binds to B and the hard reset discards B's
state. The round-13 race tests acquire the permit before introducing drift,
so they never cover this boundary.

## What
The fingerprint binds at permit acquisition (command execution), not at the
moment of user confirmation; everything that drifted before dispatch becomes
the accepted baseline.

## Approach
Bind the user-visible confirmation to the FULL merge identity (revised after
the reviewer's round-2 reopen: the `MERGE_HEAD` OID alone cannot distinguish
two merges of the same incoming commit against different local states).
`MergeState` carries an opaque `attestation` — `MERGE_HEAD` OID + the same
index/worktree fingerprint the `RecoveryPermit` binds; the conflict dialog
captures it when the conflict is surfaced, `force_resolve_merge` (UI wrapper
→ Tauri command → core) compares it against the freshly acquired permit's
bound OID AND fingerprint, failing closed on mismatch or unparsable input.
The Force button disables when no attestation was captured, and a fail-closed
refusal re-captures the live state so the dialog offers a fresh confirmation.

## Files changed
- `crates/core/src/core/git.rs` — `MergeState.attestation`;
  `force_resolve_merge` gains `expected_attestation: Option<&str>`
- `crates/core/src/core/project/fidelity.rs` — `merge_fingerprint` pub(crate);
  `RecoveryPermit::bound_fingerprint()`
- `src-tauri/src/commands/git.rs` — command + inner fn take `attestation`
- `ui/src/commands/git.ts`, `ui/src/components/revisions/Revisions.tsx` —
  capture at conflict surface, require it to enable Force, pass it through,
  re-capture on refusal

## Guard proof
- `crates/core/tests/merge_recovery.rs::force_resolve_fails_closed_when_confirmation_is_stale`
  — wrong/unparsable attestation refused, tree untouched; drilled red by
  disabling the comparison.
- `crates/core/tests/merge_recovery.rs::force_resolve_distinguishes_merges_of_the_same_incoming_commit`
  — abort + re-merge of the same draft against a different local state (same
  `MERGE_HEAD`) refused; drilled red by comparing only the OID half.

caught by the permit's fingerprint comparison at use.
