# s4-4: Stale merge-state responses cross project boundaries

**Severity**: MEDIUM — needs a project switch racing an in-flight query, but
the consequence is a hidden recovery banner (writer stranded mid-merge with no
exits) or a banner acting on the wrong project's state.
**Status**: Verified
**Branch**: n/a — pre-commit fix folded into the slice-4 commit (see s4-1).
**Commit**: (slice-4 commit)

## Evidence
`ui/src/hooks/useMergeState.ts` `refreshMergeState` applies
`gitCmd.mergeState(projectPath).then(setState)` with no cancellation or
request-generation check.

## Predicted observable failure
Project A's query resolves after the user opens project B: a late
`in_progress: false` hides B's merge banner indefinitely; a late `true`
renders a banner describing A while its Complete/Abort dispatch against B.

## What
Async responses are not tied to the path/generation that requested them.

## Approach
Two layers (revised across the reviewer's two reopens). A request-sequence
ref drops LATE responses (a query for A resolving after a switch to B). And —
because an effect-based clear still lets the first post-switch render pair
A's already-resolved state with B's path — the snapshot is stored WITH the
path that produced it and the returned state is derived at render time
(`snapshot.path === projectPath ? state : null`), so a mismatched snapshot
cannot render in any frame.

## Files changed
- `ui/src/hooks/useMergeState.ts` — sequence-guarded responses +
  path-scoped, render-time-derived snapshot

## Guard proof
- `ui/src/test/mergeBanner.test.tsx::"a stale response from a previous project
  cannot overwrite the current state"` — late response dropped; drilled red by
  removing the sequence guard.
- `ui/src/test/mergeBanner.test.tsx::"an already-resolved previous project's
  state never renders against the new path — not even pre-effect"` — a
  render-recording probe asserts no render pairs the new path with the old
  state; drilled red by removing the render-scoping.
