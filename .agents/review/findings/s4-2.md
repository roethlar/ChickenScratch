# s4-2: Recovery fingerprint omits index (staged) contents

**Severity**: HIGH — staged-but-unmerged resolution work can be hard-reset
without tripping the fail-closed drift check; plan round 13 says
"index/worktree fingerprint" and the index half was missing.
**Status**: Verified
**Branch**: n/a — pre-commit fix folded into the slice-4 commit (see s4-1).
**Commit**: (slice-4 commit)

## Evidence
`crates/core/src/core/project/fidelity.rs` `merge_fingerprint` hashes
`MERGE_HEAD` + per-path status bits + worktree bytes only.

## Predicted observable failure
Mid-merge, a file staged once and then edited (INDEX_* | WT_MODIFIED): an
external tool restages different content and restores the worktree bytes —
status bits and worktree bytes unchanged, fingerprint identical —
re-attestation passes and Force/Abort discard the newly staged work.

## What
Index-side drift between permit issue and use is invisible to the
worktree-only fingerprint.

## Approach
Hash the index entries too: every entry's path, stage, and blob OID feeds the
fingerprint, so staged-content changes (and conflict-stage changes) drift it.

## Files changed
- `crates/core/src/core/project/fidelity.rs` — `merge_fingerprint` folds in
  `repo.index()` entries (path bytes, stage flags, entry id)

## Guard proof

  — restage different content externally with worktree bytes and status bits
  unchanged; force must refuse. Reverting the index hashing makes it FAIL.
