## Verdict: REOPEN

The `onCloseRequested` plumbing is correct: handler is async, awaits the flush, `event.preventDefault()` on failure, backup remains best-effort, runtime gating prevents the beforeunload fallback from co-registering in Tauri, subscription cleanup is correct (`disposed` flag + unlisten on unmount).

But the user-visible failure path is silent.

## Concrete issue

`ui/src/App.tsx:176-178` — when `flushPendingEditorSave()` rejects:

```ts
event.preventDefault();
// (silent return)
```

There's no toast, no log, no console message. From the user's seat:

1. They click close (Cmd-W or red dot).
2. Nothing happens.
3. They assume the app is hung, click again, same silent failure.
4. They force-quit, losing the very unsaved edits the flush was trying to protect.

`toastError` is already imported at `App.tsx:39` and used elsewhere (`App.tsx:394`), so the fix is one line.

The previous broken `beforeunload` path at least let the browser show its native unsaved-changes dialog. The new `onCloseRequested` path strictly *removes* user feedback in the failure case.

## Recommended fix

`ui/src/App.tsx:176-178`: add a toast before / after `event.preventDefault()`:

```ts
event.preventDefault();
toastError(
  "Close canceled because the latest editor changes could not be saved. " +
  "Please retry, or check the editor for errors."
);
```

Add a manual repro note to the finding doc:
- Simulate a backend failure (e.g. read-only project dir) so the save fails.
- Click close.
- Confirm the toast appears and the window stays open.

## Optional nit (non-blocking)

`ui/src/App.tsx:182-186` — `.catch(() => {})` on the `onCloseRequested` registration swallows errors. If the listener never registers (Tauri API regression, permission denied, etc.), close events will tear down the window without flush. A `console.warn("onCloseRequested registration failed", e)` would aid diagnosis. Not blocking — defensive logging.

## Everything else verified

- Import path correct (`@tauri-apps/api/window`).
- `isTauriRuntime()` gates `beforeunload` to browser only; the two handlers can't co-register (App.tsx:170, 188-191).

- Backup failure independent of flush failure; flush returning `false` skips backup entirely.
