# R-12 reopened

**Branch SHA reviewed**: `b5eb5d2` (on `fix/r-12-ci-safe-macos-dmg`)
**Reviewer verdict**: REOPEN
**Reason**: pin/archive drift introduced; release source tarball checksum gate is now silently invalid.

## Issue

R-12 modifies two files that ARE included in the release source tarball (neither has `export-ignore` in `.gitattributes`):

- `.github/workflows/tauri-bundles.yml` (lines 47–48: `env: CI: true` added)
- `RELEASE.md` (line 46 + new explanatory line)

After applying the patch on top of master (R-11 verdict `0a2151e`), running `scripts/create-release-source.sh 1.0.0 HEAD` now produces:

```
f9e06fea47702e4aea64bac6c062ee013ff6208c9c828a67c00389bf61b47125
```

But `pkg/arch/PKGBUILD` still pins R-11's pre-R-12 archive checksum:

```
sha256sums=('a3f63fde0eabcc423c765e1afdad4eb0859930aa08acbdc4eedb94fb69253334')
```

**Concrete consequence**: when the human cuts `v1.0.0` from this state, Stage 5 of `RELEASE.md` (`scripts/create-release-source.sh "$version" "v$version" dist` followed by `--require-tag` confirmation) will produce a tarball whose SHA no longer matches the pin, and any consumer running `makepkg --verifysource` on the Arch package would also fail with a checksum mismatch.

`scripts/check-release-metadata.sh --release 1.0.0` does NOT catch this — it only verifies the pin's shape (`SKIP`/64-char hex), not equality against the actual archive bytes. That's a separate latent gap (see Follow-up below) but is the reason this drift wasn't surfaced by the existing gate.

## Required fix

Update `pkg/arch/PKGBUILD` line containing `sha256sums=(...)`:

```diff
-sha256sums=('a3f63fde0eabcc423c765e1afdad4eb0859930aa08acbdc4eedb94fb69253334')
+sha256sums=('f9e06fea47702e4aea64bac6c062ee013ff6208c9c828a67c00389bf61b47125')
```

Then re-verify on the branch:

```bash
tmpdir=$(mktemp -d)
scripts/create-release-source.sh 1.0.0 HEAD "$tmpdir" | grep -oE '[a-f0-9]{64}' | head -1
# expect: f9e06fea47702e4aea64bac6c062ee013ff6208c9c828a67c00389bf61b47125
grep -oE '[a-f0-9]{64}' pkg/arch/PKGBUILD | head -1
# expect: same SHA
rm -rf "$tmpdir"
scripts/check-release-metadata.sh --release 1.0.0
# expect: "release metadata ok for 1.0.0"
```

## Follow-up (separate finding, not blocking this reopen)

`scripts/check-release-metadata.sh` should also verify that the pinned `sha256sums` value equals the actual `scripts/create-release-source.sh` output for the same version. Otherwise the pin can silently drift any time a tarball-included file changes after pinning. Worth a separate `R-NN` finding once R-12 lands.
