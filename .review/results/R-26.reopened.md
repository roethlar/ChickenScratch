**Branch SHA reviewed**: `b14476b93591559c687991a402cfd5b1641ca882` on `fix/r-26-validation-path-filter`
**Reviewer verdict**: REOPEN
**Reason**: R-26 modifies `.github/workflows/validation.yml`, which is inside the release tarball (no `export-ignore` in `.gitattributes`), but does not re-pin `pkg/arch/PKGBUILD`. The R-13 gate that R-26 itself is strengthening now fails on master after merge — `create-release-source.sh 1.0.0 HEAD` produces `2f3da59cdb967ee39404a42db3dc6f7974b3c9de74dd8b0804154bf93951c6fa` but the pin is still `4af64eb…ddeb47`.

This is the **second occurrence** of this exact pattern (R-12 had it too). Worth adding to GPT's pre-commit checklist whenever modifying a non-`export-ignore` file:

```bash
tmpdir=$(mktemp -d)
GEN=$(scripts/create-release-source.sh 1.0.0 HEAD "$tmpdir" | grep -oE '[a-f0-9]{64}' | head -1)
PIN=$(grep -oE '[a-f0-9]{64}' pkg/arch/PKGBUILD | head -1)
[[ "$GEN" == "$PIN" ]] && echo "PIN OK" || echo "RE-PIN: $GEN"
rm -rf "$tmpdir"
```

Or just run `scripts/check-release-metadata.sh --release 1.0.0` before pushing — R-13 makes it a one-line check that catches exactly this drift.

## Forward-looking: should the gate be made impossible to forget?

Two options worth considering (not blocking this reopen):

1. **Pre-commit hook**: a one-liner in `.git/hooks/pre-commit` or `.githooks/` that runs `scripts/check-release-metadata.sh` and blocks commits with stale pins. Local-only, fast, opt-in.
2. **Auto-update the pin** in `scripts/check-release-metadata.sh` with a `--fix` flag, so the workflow becomes "make your change, then run the gate, paste the new SHA into PKGBUILD." Less magical than auto-commit, more discoverable.
