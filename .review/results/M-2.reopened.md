## Verdict: REOPEN

The `.meta` fail-loud + pre-flight validation work is correct and ship-ready. The `writing-history.json` write/read path consolidation is correct. But there's a **third** read path on the same data file that wasn't updated, contradicting your own writeup ("read and write paths share the same parser").

## Concrete issue

`src-tauri/src/commands/io.rs:533-536` — `get_session_progress` reads `writing-history.json` with:

```rust
fs::read_to_string(&history_path)
    .ok()
    .and_then(|s| serde_json::from_str(&s).ok())
    .unwrap_or_default()
```

This is the exact silent-default shape M-2 was supposed to retire. The other two callsites (the strict-parse helper `parse_writing_history` at `io.rs:444-452` plus its uses at `:440` and `:470`) are correctly tightened; this one was missed.

## Why it matters (even though it's a read path)

- A user inspecting "Session Progress" with a corrupt history file gets a silent zero today-words count instead of a clear error banner. That's misleading, not destructive — but it directly contradicts the invariant claimed in your `.review/findings/M-2.md`: "Replaced `writing-history.json` `unwrap_or_default()` parsing with a strict JSON parser shared by read and write paths."
- The two paths now disagree on what "valid" means: `record_daily_words` will refuse to write into a corrupt file (good), but `get_session_progress` will keep happily reporting zeros (bad signal hygiene). A user could see "0 words today" in the UI, write a chapter, then watch the save fail with a corruption error from a *different* code path that hits the strict parser.

## Recommended fix

`src-tauri/src/commands/io.rs:533-536`: replace the silent-default block with a call to the shared `parse_writing_history` helper. Single-line change; same `WritingHistory` return type; no other call sites affected.

Add a test for the `get_session_progress` corruption path that asserts it propagates the parse error rather than returning a default-zero result.
