# M-4 reopened — shell.open regex missing anchors

**Branch**: `fix/m-4-tauri-csp-shell-scope`
**Commit reviewed**: `bce74b5`
**Reviewer**: claude

## Verdict: REOPEN

CSP work is solid (prod-strict, dev-relaxations isolated, capability change to `shell:default` is safe in plugin v2.3.5). But the `plugins.shell.open` regex regression-tests open.

## Concrete issue

`src-tauri/tauri.conf.json:39` — the regex value for `plugins.shell.open` is **missing `^` start anchor and `$` end anchor**.

`tauri-plugin-shell` v2.3.5 validates via `regex.is_match(path)` at `scope.rs:210`, which is **substring** mode. Then it opens the **original unmodified path** via `::open::that_detached(path)` at `scope.rs:228` — not the regex-matched substring.

This means the attack class M-4 was supposed to block remains:

- `file:///Users/victim/.ssh/id_rsa#https://x` → `is_match` returns true (matches `https://x` substring), `open::that_detached` dispatches the `file://` URL to the OS default handler.
- `javascript:eval('...')//https://x` → same shape.
- `vscode://...#https://x`, custom-handler-protocol abuse → same shape.

The hostile-`.chikn` threat model that M-4 was designed to close — manuscript content injecting a malicious link that, when clicked, reaches an OS handler — is still reachable.

## Why Tauri's default does this right

Tauri's documented default for `plugins.shell.open` is:

```
^((mailto:\w+)|(tel:\w+)|(https?://\w+)).+
```

Note the leading `^`. Without it, an attacker prefixes their malicious URL with any allowed-scheme substring and bypasses the check.

## Recommended fix

Change `src-tauri/tauri.conf.json:39` to anchor both ends:

```
^https://[A-Za-z0-9][A-Za-z0-9.-]*(?::[0-9]{1,5})?(?:[/?#][^\s]*)?$
```

Then verify both existing call sites still pass:
- `ui/src/App.tsx:370` — `https://github.com/...` (should match)
- `ui/src/components/welcome/Welcome.tsx:94` — `https://pandoc.org/...` (should match)

Add a unit test (or at least a documented manual check) that exercises the bypass shape and confirms rejection:
- `file:///etc/passwd#https://x` — must reject
- `javascript:x//https://y` — must reject
- `https://example.com/path?q=1#frag` — must accept
