## R-19: Native readers still persist destructive repair/malformed document loss

### Approach

- Swift `TreeNode` now retains the document `path` from `project.yaml`, so a hierarchy entry whose `.md` file is temporarily missing can be read and written back without dropping the binder node.
- Swift document loading now fails closed for missing `.meta`, corrupt/non-mapping `.meta`, missing metadata `id`, and unreadable `.md` content instead of silently dropping the document or replacing content with an empty string.
- Swift project writing now rejects document hierarchy nodes that cannot resolve to a safe path from either `project.documents` or the retained hierarchy path, preventing `project.yaml` rewrites with null or empty document paths.
- Windows reader repair no longer removes hierarchy document nodes or document map entries when the referenced `.md` file is missing; it validates hierarchy paths but preserves the manifest.
- Windows document reads now fail if a discovered `.md` cannot be read, and Windows project writing validates document paths in the hierarchy before serializing `project.yaml`.

### Known Gaps

- Swift uses fail-closed malformed metadata handling for this slice rather than Rust's corrupt `.meta` quarantine path. That prevents destructive native rewrites, but exact quarantine behavior is still not shared across languages.
- Linux native delete/repair behavior remains tracked separately in R-20.
