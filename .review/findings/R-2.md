# R-2: Windows CI workflow points at stale solution and SDK

## Approach

Updated the Windows GitHub Actions workflow to install the .NET 10 SDK required by the Windows projects' `net10.0` and `net10.0-windows10.0.19041.0` targets.

Changed restore/build commands from the missing `ChickenScratch.sln` file to the checked-in `ChickenScratch.slnx`, matching the README and repository layout.

Removed the invalid solution-level `/p:Platform=x64` override. The `.slnx` has project-specific platform metadata for the WinUI app, while the core and harness projects use the default platform. Passing `Platform=x64` at solution scope fails with `MSB4126`.

Expanded the final CI step to build both checked-in core test harness projects instead of rebuilding only the core library.

## Tests

- `cd windows && dotnet restore ChickenScratch.sln` (expected failure on old path: missing solution)
- `cd windows && dotnet restore ChickenScratch.slnx /p:Platform=x64` (expected failure on old platform override: invalid solution configuration)
- `cd windows && dotnet restore ChickenScratch.slnx /p:EnableWindowsTargeting=true`
- `cd windows && dotnet build ChickenScratch.Core/ChickenScratch.Core.csproj /p:Configuration=Release --no-restore`
- `cd windows && dotnet build ChickenScratch.Core.Tests/GitServiceRestoreHarness/ChickenScratch.Core.GitServiceRestoreHarness.csproj /p:Configuration=Release --no-restore`
- `cd windows && dotnet build ChickenScratch.Core.Tests/CrossFrontendHarness/ChickenScratch.Core.CrossFrontendHarness.csproj /p:Configuration=Release --no-restore`

## Known gaps

Full WinUI solution build still requires a Windows host because the Windows App SDK XAML compiler is a Windows executable. On macOS, `dotnet build ChickenScratch.slnx /p:Configuration=Release /p:EnableWindowsTargeting=true --no-restore` restores and compiles the non-WinUI projects, then fails when it tries to execute `XamlCompiler.exe`.
