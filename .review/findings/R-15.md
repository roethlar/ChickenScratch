# R-15 Atomic Document Writes

## Approach

Rust core document persistence now uses a single same-directory atomic write helper for `project.yaml`, non-empty `threads.yaml`, document `.md`, and document `.meta` files. The helper stages bytes in the target directory, preserves existing permissions when replacing an existing file, syncs the staged file, atomically persists it over the destination on Unix and Windows through `tempfile`, syncs the containing directory on Unix, and lets failed staging/persist clean up the temp file instead of truncating the destination.

Rust `write_document` also serializes metadata before replacing the content file, so a serialization failure leaves both the existing `.md` and `.meta` untouched. Existing path traversal and symlink validation still runs before any temp file is staged.

Rust reader recovery now quarantines a corrupt per-document `.meta` sidecar to `*.meta.corrupt-<uuid>` and continues loading the document with default metadata. When the hierarchy already references that document path, the fallback preserves the hierarchy id/name so the binder remains coherent and `project.yaml` is not rewritten during recovery. Corrupt `threads.yaml` and corrupt `project.yaml` still fail loudly.

The Windows native writer now stages `.md` and `.meta` sidecars into same-directory temp files before replacing/moving them into place, with cleanup in `finally`. The Windows cross-frontend harness gained an `--atomic-writes` mode covering existing-file replacement, new-file creation, sidecar creation, and temp cleanup. Swift already uses atomic writes for these surfaces, so it is unchanged.

## Known gaps

This is per-file atomic replacement, not a true multi-file transaction across `.md` and `.meta`; filesystems do not provide a portable two-file commit primitive here. The branch stages and validates before replacement to shrink the failure window, and avoids the original truncate-in-place data-loss mode. Native malformed `.meta` convergence for Swift/Windows remains tracked under R-19.
