## R-27: Rust release/package builds do not enforce Cargo.lock

### Approach

- Added `cargo metadata --locked --format-version 1` to the release metadata gate so stale Rust dependency resolution fails during the existing validation preflight.
- Made validation CI run clippy and tests with `--locked`.
- Added locked Rust dependency preflight steps before macOS and Linux Tauri bundle builds.
- Passed `--locked` through Tauri CLI bundle builds using trailing Cargo args (`cargo tauri build ... -- --locked`).
- Made the Arch package build use `cargo build --locked`.
- Updated the release runbook to document the locked metadata preflight and locked Cargo/Tauri build commands.

### Tests

- `cargo metadata --locked --format-version 1 >/dev/null`
- `scripts/check-release-metadata.sh --release 1.0.0`
- `cargo fmt --all -- --check`
- `cargo clippy --locked -p chickenscratch-core -p chickenscratch -p chickenscratch-tui -p chikn-converter --all-targets -- -D warnings`
- `cargo test --locked -p chickenscratch-core -p chickenscratch -p chickenscratch-tui -p chikn-converter --lib --bins --tests`
- `npm run lint && npm run build` in `ui`

### Files Changed

### Known Gaps

- Native macOS/Linux Tauri bundle builds and Arch `makepkg` validation still need their respective CI/native hosts; the branch enforces locked Cargo resolution in those commands but does not produce those artifacts locally.
- The first locked full Rust suite attempt hit the existing flaky `utils::process::tests::process_helper_caps_combined_stdout_and_stderr`; the immediate rerun passed the full locked suite.
