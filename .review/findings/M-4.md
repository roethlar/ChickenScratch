# M-4: Tauri CSP and shell.open scope

## Approach

- Replaced the disabled Tauri CSP with a production policy that defaults to same-origin, explicitly allows Tauri IPC (`ipc:` and `http://ipc.localhost`), local app assets, data images/fonts, inline styles, and blocks object/embed and framing surfaces.
- Added a separate dev CSP for the configured Vite dev server at `http://localhost:1420`, including the Vite HMR websocket and development script/style allowances.
- Replaced unconstrained `plugins.shell.open: true` with an anchored HTTPS-only URL validator. The regex requires `https://`, a host-shaped first component, and only optional path/query/fragment suffixes; prefix/suffix bypasses like `file://...#https://x` do not match.
- Switched the capability entry from `shell:allow-open` to `shell:default`; the exposed renderer command remains `open`, with the tighter URL validator enforced by the shell plugin config.
- Added a regression test that reads `tauri.conf.json` and checks accepted HTTPS URLs plus rejected `http`, `file`, `javascript`, and custom-handler prefix-bypass shapes.

## Known gaps

- The shell URL constraint is regex-based because `tauri-plugin-shell` v2.3.5 validates `open` paths with a configured regex rather than a parsed URL policy.
- The dev CSP intentionally includes `script-src 'unsafe-eval'` and localhost Vite websocket access for the development server only.
