# R-29: macOS DMG release path has no signing/notarization gate

## Approach

The existing macOS bundle workflow remains a fast unsigned smoke build, but its job and artifact names now state that the artifacts are smoke-only. Public macOS distribution now has a dedicated `macOS Signed Release` workflow triggered by `v*` tags or `workflow_dispatch` and gated by the protected `release-macos` environment.

The release workflow fails before building if any Apple signing/notarization secret is missing, imports the Developer ID Application certificate into a temporary keychain, writes the App Store Connect API key to `APPLE_API_KEY_PATH`, and builds the app/DMG through the same locked Tauri command used by the smoke path. It then explicitly submits and staples the DMG with `xcrun notarytool` before uploading artifacts only after `codesign`, `spctl`, and `stapler` verification passes for the app and DMG.

`tauri.conf.json` now explicitly declares hardened runtime for macOS bundles. The release runbook documents the unsigned-vs-signed artifact split, required secrets, and mandatory verification commands. The roadmap no longer presents macOS signing/notarization automation as future work.

## Known Gaps

The full signed/notarized path requires Apple Developer credentials and therefore is enforced in the protected GitHub Actions release job rather than locally. No app-sandbox entitlement is added; this is a Developer ID distribution path with hardened runtime, and the app needs user-selected filesystem/project access outside the Mac App Store sandbox model.
