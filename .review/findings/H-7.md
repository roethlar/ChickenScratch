# H-7: Stale-disk-state on restore/compile/file-history

Branch: `fix/h-7-flush-before-disk-actions`

## Approach

- `DocumentHistory` now flushes any pending editor save before loading per-document history, so the history view is based on the latest editor buffer that has reached disk.
- Document restore now flushes before invoking `restoreDocument`, reloads the project through the store's `setProject` action, and explicitly sets the mounted editor content to the restored active document content when the restored document is still active.
- Added `setCurrentEditorMarkdown` beside the existing editor ref helpers so restore can replace the TipTap buffer with `emitUpdate: false` instead of relying on same-id selection effects.
- `CompileDialog` now flushes pending editor saves before `compileProject`, so compile/export reads the latest active editor content from disk.

## Tests

- No focused UI test was added because this UI package currently has lint/build scripts but no test runner configured.
- Validated with `cd ui && npm run lint`.
- Validated with `cd ui && npm run build`.

## Manual behavior

- Open file history after typing in the active editor: pending debounced edits are flushed before history is read.
- Restore the currently active document from file history: pending edits flush first, the project reloads, and the visible editor buffer is replaced with the restored content so a stale debounce cannot overwrite it.
- Export/compile after typing in the active editor: pending debounced edits flush before the backend compile reads project files.
