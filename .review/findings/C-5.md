# C-5: AI streaming writes to wrong document after navigation

## Approach

- Added a Tauri-managed `AiStreamRegistry` keyed by stream request id.
- Added `cancel_ai_transform_stream` so the frontend can cancel a specific stream.
- Registered stream ids before handling stream input, including empty-input streams, so a cancel issued before backend registration is remembered.
- Threaded cooperative cancellation checks through Ollama, Anthropic, and OpenAI streaming before work, between stream reads, and before chunk emission.
- Stamped each toolbar AI stream with the active project path plus a context key: `doc:<activeDocId>` for normal editing or `flow:<ordered doc ids>` for Flow mode.
- Checked the live `useProjectStore.getState()` project/context key before every chunk/done/error event is applied.
- Canceled and locally aborted the stream when navigation changes the active document/project, preventing late chunks from writing into the reused Tiptap editor instance.

## Tests

- Added Tauri unit tests for canceling a registered stream and remembering a cancellation before registration.
- UI has no focused Tiptap stream navigation test harness in this repo; validation is via lint/build.

## Known gaps

- Cancellation is cooperative. If a provider read is blocked, the thread may not exit until another line arrives or the HTTP timeout fires, but no further events are emitted after the cancellation flag is observed.
- No automated UI harness exists here to simulate Tiptap stream chunks across document navigation.
