# H-6: Real Cross-Frontend Harness

## Approach

- Added `crates/core/tests/cross_frontend/run.sh` as the script-driven harness.
- The harness builds and runs `chikn-converter` against `samples/Corn.scriv`, using a test-only Pandoc shim so the fixture conversion works on machines without Pandoc installed.
- After each pass, the harness invokes the Rust reader through an env-driven integration test in `cross_frontend_round_trip.rs`.
- Added `ChiknKitCrossFrontendHarness`, a Swift executable target that opens a `.chikn` project with `ChiknKit.Reader`, applies a deterministic metadata/fields mutation through `ChiknKit.Writer.saveDocumentMeta`, and saves.
- Added `ChickenScratch.Core.CrossFrontendHarness`, a C# console harness that opens a `.chikn` project with `ProjectReader`, applies a deterministic metadata/fields mutation, and saves through `ProjectWriter`.
- Changed `windows/ChickenScratch.Core` from `net10.0-windows` to `net10.0` so the core library and console harness can build on macOS without WinUI targeting. The WinUI app remains on its existing Windows target.

## Known Gaps

- The harness records a manifest and verifies that Rust can load after each writer pass instead of byte-for-byte golden comparison. Current writers update timestamps and may repair or reorder project metadata, so byte-for-byte goldens would be brittle without a larger normalization layer.
- The current Swift/C# writer passes cause the Rust reader to log in-memory hierarchy repair on the converted Scrivener fixture. The harness exposes that drift in command output and still verifies the marker fields, but it does not yet fail on repair logs. Tightening this into a no-repair/golden assertion should be a follow-up once the current frontend writer drift is fixed or normalized.
- The script skips Swift or C# only when the corresponding toolchain is missing. Rust converter remains mandatory.
