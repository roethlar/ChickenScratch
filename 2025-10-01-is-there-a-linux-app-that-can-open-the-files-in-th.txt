● 📋 Legal & Format Strategy

  The Scrivener Format Situation:

  Good news: Reverse engineering file formats is generally legal for interoperability (US fair use doctrine)

  Gray area: Scrivener's EULA prohibits reverse engineering, but:
  - This is a contractual restriction (not copyright law)
  - Only binds people who agreed to the EULA
  - You can analyze .scriv files you already own
  - Clean-room implementation (describe format → someone else codes) is typically safe

  Reality check: Many apps read/write proprietary formats (LibreOffice ↔ .docx, GIMP ↔ .psd). As long as you're not copying Scrivener's code, you're likely
  fine.

  Recommendation: Build with .scriv compatibility, but have a fallback if challenged.

  ---
  🎯 Proposed Format Strategy

  Native Format: "Manuscript Project" (.msproj)

  A clean-room designed format that:
  - Git-optimized: Plain text where possible (YAML metadata, Markdown/RTF hybrid)
  - Lossless .scriv round-tripping: Import .scriv → .msproj → export .scriv with zero data loss
  - Future-proof: Open spec, extensible, survives even if Scrivener disappears

  Structure:
  MyNovel.msproj/
  ├── project.yaml              # Metadata, settings, hierarchy
  ├── manuscript/
  │   ├── ch01-opening.md      # Markdown or RTF per-document choice
  │   ├── ch02-conflict.md
  │   └── .meta/               # Per-document metadata (labels, status, synopsis)
  │       ├── ch01-opening.yaml
  │       └── ch02-conflict.yaml
  ├── research/                # Research folder
  │   └── character-notes.md
  ├── templates/               # Character/setting templates
  ├── settings/                # Compile settings, themes
  └── .git/                    # Git integration native

  Benefits:
  - Writer-friendly: Human-readable, editable in any text editor
  - Git-native: Text diffs work perfectly, merge conflicts are manageable
  - AI-friendly: Simple, predictable structure for Claude to edit
  - Extensible: Add new features without breaking old projects

  Migration path:
  1. Import .scriv → convert to .msproj (one-time, lossless)
  2. Work in .msproj → all editing, git workflows
